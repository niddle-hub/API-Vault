# Спецификация для код-агента: Локальный менеджер API-ключей

## 1. Общая концепция

Веб-приложение, запускаемое локально на компьютере пользователя, предназначенное для безопасного хранения API-ключей различных сервисов. Ключи хранятся в зашифрованном виде в базе данных. Доступ к ключам открывается после ввода мастер-пароля. Основной интерфейс – таблица со списком ключей, с возможностью копирования, редактирования и удаления.

## 2. Функциональные требования

### 2.1 Аутентификация
- При первом запуске или после сброса сессии пользователь попадает на страницу входа.
- Поле для ввода мастер-пароля.
- Мастер-пароль используется для генерации ключа шифрования (PBKDF2 с солью).
- Соль хранится в конфигурации/базе.
- Успешный вход создаёт серверную сессию, в которой сохраняется ключ шифрования (не сам пароль).
- Сессия действительна до перезапуска сервера или истечения тайм-аута (настраиваемый, например, 30 минут бездействия).

### 2.2 Главная страница (список ключей)
- Таблица/список всех сохранённых записей, отсортированных по дате создания (или по названию).
- Каждая строка содержит:
  - **Название** сервиса/платформы (текст).
  - **Замаскированное значение ключа**: показываются первые 4 символа, затем `****`, затем последние 4 символа (если длина ключа >= 8). Для коротких ключей – только звёздочки.
  - Кнопка **Копировать** (иконка или текст) – копирует настоящий (расшифрованный) ключ в буфер обмена. После нажатия кратковременно появляется уведомление «Скопировано».
  - Кнопка **Редактировать** – открывает модальное окно/строчный режим редактирования (см. 2.4).
  - Кнопка **Удалить** – запрашивает подтверждение и удаляет запись.

### 2.3 Добавление нового ключа
- Под списком находится строка ввода с двумя полями: **Название** и **Ключ**.
- Поле «Ключ» может иметь кнопку «глаз» для показа/скрытия вводимого текста.
- Кнопка **Добавить** сохраняет запись: ключ шифруется на сервере и сохраняется в БД.
- После успешного добавления список обновляется без перезагрузки страницы.

### 2.4 Редактирование существующего ключа
- Открывается модальное окно или форма прямо в строке с полями Название и Ключ.
- Поле Ключ может быть предварительно заполнено маскированным значением; при фокусе показывается реальный ключ (расшифрованный), либо пользователь может кликнуть «Показать/скрыть».
- Кнопка **Сохранить** – отправляет изменённые данные (ключ заново шифруется) и закрывает форму.
- Кнопка **Отмена** – закрывает без изменений.

### 2.5 Копирование ключа
- При нажатии кнопки «Копировать» выполняется асинхронный GET-запрос на сервер для получения расшифрованного значения.
- Сервер расшифровывает ключ текущим сессионным ключом шифрования и возвращает plaintext.
- Клиент копирует значение в буфер обмена (`navigator.clipboard.writeText`).
- Зашифрованное значение никогда не передаётся на клиент, кроме как через замаскированное представление.

## 3. Нефункциональные требования

- **Безопасность**:
  - Все ключи в БД зашифрованы симметричным алгоритмом (AES-256-GCM или Fernet).
  - Мастер-пароль никогда не сохраняется на диске.
  - Ключ шифрования хранится только в оперативной памяти сервера во время сессии.
  - Защита от XSS, CSRF (при необходимости).
  - Использование HTTPS не требуется, так как приложение локальное (http://localhost:<port>).
- **Локальное исполнение**: Приложение запускается одной командой, порт задаётся в конфигурации (по умолчанию 5000).
- **Переносимость**: Данные (SQLite база) и конфигурация находятся в одной папке с приложением.

## 4. Технологический стек

- **Backend**: Python 3.10+, Flask
- **Шифрование**: библиотека `cryptography` (Fernet) – ключ Fernet генерируется из мастер-пароля через PBKDF2HMAC (соль, 100 000 итераций).
- **База данных**: SQLite (файл `keys.db`), содержащая таблицу `keys` с полями `id`, `name`, `encrypted_value`, `created_at`. (encrypted_value – бинарное поле для Fernet-токена).
- **Сессии**: Flask-Session с файловым или серверным хранилищем (файлы в папке `flask_session`).
- **Frontend**: Jinja2-шаблоны, ванильный JavaScript (или минимальное использование HTMX/Alpine.js для интерактивности). Bootstrap (опционально) для стилей.
- **Сборка и запуск**: `requirements.txt`, запуск через `python app.py`.

### 5.2 Инициализация и шифрование
- При старте, если БД не существует, создаётся таблица `keys`.
- Настройки шифрования: соль для PBKDF2 генерируется один раз и сохраняется в файл (например, `salt.bin`) или в `config.py`. Либо хранить соль прямо в БД.
- Функция `derive_key(master_password: str, salt: bytes) -> bytes` возвращает ключ для Fernet (32 байта URL-safe base64). Fernet требует ключ в base64, поэтому после PBKDF2 получаем 32 байта и кодируем.
- Шифрование: `f = Fernet(key); encrypted = f.encrypt(plaintext.encode())`.
- Дешифрование: `f.decrypt(encrypted).decode()`.

### 5.3 API эндпоинты

| Метод | Путь                    | Описание |
|-------|-------------------------|----------|
| GET   | `/`                     | Перенаправляет на `/login` если нет сессии, иначе на `/keys` |
| GET/POST | `/login`            | Форма входа; POST проверяет пароль и создаёт сессию с ключом |
| GET   | `/keys`                 | Главная страница со списком (возвращает шаблон `index.html`) |
| GET   | `/api/keys`             | JSON-список всех ключей (id, name, masked_value) |
| POST  | `/api/keys`             | Создать новый ключ (JSON `{name, value}`) – сервер шифрует value и сохраняет |
| GET   | `/api/keys/<id>/value`  | Возвращает расшифрованное значение (требуется сессия) |
| PUT   | `/api/keys/<id>`        | Обновить имя и/или значение (JSON `{name?, value?}`). Если передан value, перешифровывается. |
| DELETE| `/api/keys/<id>`        | Удалить запись |
| POST  | `/logout`               | Очистить сессию |

### 5.4 Фронтенд поведение (index.html)
- При загрузке страницы делается fetch на `/api/keys`, рендерится таблица.
- Кнопка «Копировать»: вызывает `/api/keys/<id>/value`, получает plaintext, копирует в буфер, показывает всплывающее уведомление.
- Кнопка «Редактировать»: открывает модальное окно, загружая текущие данные (можно сделать ещё один запрос на `/api/keys/<id>` для получения полного имени и маскированного ключа, или использовать данные из таблицы). При сохранении отправляет PUT с обновлёнными данными.
- Кнопка «Удалить»: подтверждение через `confirm()`, затем DELETE, обновление списка.
- Строка добавления: после ввода названия и ключа, отправка POST, обновление списка.

### 5.5 Маскирование ключа
- На сервере (или на клиенте) создаётся `masked_value`: если длина > 8, `original[:4] + '****' + original[-4:]`; иначе `'****'`. Это маскированное значение сохранять не нужно, только вычислять при выводе списка.

## 6. Пошаговый план реализации (для агента)

1. Создать структуру проекта, установить зависимости (Flask, cryptography, flask-session).
2. Реализовать `crypto_utils.py`:
   - `generate_salt() -> bytes`
   - `derive_key(password, salt) -> bytes`
   - `encrypt_value(key, plaintext) -> bytes`
   - `decrypt_value(key, encrypted) -> str`
3. Создать `config.py`: SECRET_KEY для Flask, путь к БД, SALT (сгенерированный один раз) или генерация при первом запуске.
4. В `app.py`:
   - Инициализировать Flask, настроить сессии.
   - Создать/проверить БД с таблицей `keys(id INTEGER PRIMARY KEY, name TEXT, encrypted_value BLOB, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP)`.
   - Реализовать декоратор `@login_required`, проверяющий наличие в сессии ключа `encryption_key`.
   - Эндпоинт `/login` (GET – форма, POST – принять пароль, проверить (если ещё нет ключей, любой пароль создаёт новую базу), вычислить ключ, сохранить в сессию, перенаправить на `/keys`).
   - Эндпоинты API (как в таблице) с проверкой сессии.
   - Для `/api/keys` возвращать JSON со списком `[{"id":..., "name":..., "masked_value":...}]`.
   - При старте если БД пуста, можно предусмотреть создание мастера-пароля (как первый вход). Упростим: мастер-пароль всегда хранится в голове пользователя, проверять не с чем, просто любая введённая строка считается валидной и используется как ключ шифрования. (В реальности нужно хранить хеш пароля для проверки, но для локального менеджера допустимо без хеша, иначе при неверном пароле расшифровка даст мусор. Лучше сохранить хеш для валидации. Добавим таблицу `config` с полем `password_hash` и `salt`. При первом входе создаём хеш, потом сверяем.)
5. Создать шаблоны:
   - `login.html` – простой дизайн, поле пароля, кнопка.
   - `index.html` – контейнер для списка, кнопка добавления, модальное окно для редактирования, скрипты.
6. Написать клиентский JavaScript (`static/script.js`) для динамического взаимодействия, обновления таблицы, обработки кликов.
7. Добавить минимальные CSS стили для аккуратного отображения.

## 7. Безопасность (заметки)
- Ключ шифрования в сессии хранится как байтовая строка или base64. Flask-Session по умолчанию хранит сессии в зашифрованных куках, но лучше использовать серверное хранилище (файловая система) с flask-session, чтобы не передавать чувствительные данные на клиент. Настройка `SESSION_TYPE = 'filesystem'`.
- При получении пароля через POST используйте HTTPS только если приложение проксируется, для локального хоста не критично.
- Кнопка копирования не должна сохранять расшифрованный ключ в DOM или глобальных переменных; после вставки в буфер обмена значение должно быть удалено (JS переменная выйдет из области видимости).

## 8. Ожидаемый результат
Работающее локальное веб-приложение, которое:
- Запускается командой `python app.py`.
- Открывается в браузере по адресу `http://localhost:5000`.
- Требует ввод мастер-пароля.
- Позволяет добавлять, просматривать, копировать, редактировать и удалять зашифрованные API-ключи.
